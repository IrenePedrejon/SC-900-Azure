Punto 2.1

Qué es Microsoft Entra ID

Microsoft Entra ID (antes llamado Azure Active Directory / Azure AD) es el servicio de Microsoft en la nube que gestiona identidades y accesos. Es el corazón de todo este módulo: cualquier cosa que hagas en Microsoft 365 o Azure — iniciar sesión, que te bloqueen un acceso sospechoso, que un administrador te asigne un rol — pasa por Entra ID en algún momento.

Piensa en Entra ID como el "portero + recepcionista + libro de registro" de todos los servicios de Microsoft en la nube: comprueba quién eres (autenticación), decide a qué puedes entrar (autorización), y lleva un registro de todo (auditoría).
Nota sobre el nombre: Microsoft renombró "Azure Active Directory" a "Microsoft Entra ID" en 2023. Es habitual ver ambos nombres en documentación, cursos o exámenes — son lo mismo. "Entra" es en realidad el nombre de toda una familia de productos de identidad de Microsoft (Entra ID es solo uno de ellos, aunque el más importante para este examen).

Punto 2.2b

Identidad híbrida

Ya vimos en el Módulo 1 que muchas empresas usan Active Directory local y Entra ID a la vez, sincronizados con Entra Connect. A esta situación se le llama identidad híbrida: la misma cuenta de usuario "vive" en los dos sitios a la vez, sincronizada, en vez de tener que mantener dos cuentas separadas.

Formas de autenticar una identidad híbrida

Cuando un usuario con identidad híbrida inicia sesión en Entra ID, hay tres formas distintas de comprobar su contraseña, según cómo la empresa haya configurado Entra Connect:

MétodoCómo funciona
Sincronización de hash de contraseña (Password Hash Sync)Entra Connect sincroniza una versión cifrada (hash) de la contraseña hacia la nube. Entra ID puede validar el inicio de sesión por sí solo, sin consultar al Active Directory local en cada intento.
Autenticación de paso a través (Pass-through Authentication)La contraseña nunca se sincroniza a la nube. Cada vez que alguien inicia sesión, Entra ID valida la contraseña consultando directamente al Active Directory local en tiempo real.
Federación (con AD FS)Entra ID ni siquiera valida la contraseña — delega por completo esa comprobación al Active Directory local mediante una relación de confianza, tal como vimos en el punto 1.10.

PTA vs. Federación: la diferencia que de verdad importa

PTA y Federación se parecen mucho a primera vista, porque las dos consultan al Active Directory local en tiempo real en vez de sincronizar nada por adelantado. La diferencia real no está en "si consultan al AD local", sino en dónde escribes tu contraseña:

Con PTA, es como pagar en una tienda con tarjeta: pasas la tarjeta por el datáfono de la tienda (la pantalla normal de inicio de sesión de Microsoft), y ese datáfono llama al banco (tu Active Directory local) para preguntar "¿esta tarjeta es válida?". Tú nunca hablas directamente con el banco, pero el banco sí participa en la comprobación, en tiempo real.

Con Federación, en vez de pagar en la tienda, te mandan directamente al mostrador del banco a identificarte. Entra ID ni siquiera te muestra su propio formulario de inicio de sesión: te redirige por completo a una página de tu propia empresa (el servidor AD FS), donde escribes tu contraseña. AD FS te entrega un token firmado que certifica quién eres, y ese token es lo único que Entra ID llega a ver — nunca tu contraseña.
Pass-through AuthenticationFederación
¿Dónde escribes la contraseña?En la pantalla normal de inicio de sesión de MicrosoftEn la pantalla de tu propia empresa (AD FS), a la que Entra ID te redirige
¿Qué recibe Entra ID de vuelta?Un simple "sí, es correcta" / "no, es incorrecta"Un token firmado digitalmente con más información
¿Ve Microsoft la contraseña en algún momento?Sí, brevemente, antes de descartarlaNunca, ni un instante
🎯 Truco de examen: si la pregunta dice "el usuario nunca ve ninguna pantalla de Microsoft, se le redirige a un sitio propio de su empresa" → Federación. Si dice "el usuario usa la pantalla normal de Microsoft, pero la contraseña se valida contra el AD local" → PTA. Además, con Password Hash Sync, Entra ID puede seguir validando inicios de sesión aunque el Active Directory local esté caído (porque ya tiene el hash sincronizado). Con PTA y con Federación, si el Active Directory local no está disponible, los usuarios no pueden iniciar sesión — porque la validación depende de consultarlo en el momento.

Punto 2.3

Usuarios y grupos en detalle

Tipos de usuario en Entra ID

  • Usuario miembro (Member): un empleado normal de la organización, con una cuenta creada dentro de ese mismo tenant.
  • Usuario invitado (Guest): alguien externo a la organización (un proveedor, un cliente, un colaborador puntual) al que se le da acceso limitado a ciertos recursos, sin crearle una cuenta completa de empleado. Su identidad "real" sigue viviendo en su propia organización o cuenta personal — Entra ID solo le da un acceso controlado a lo que necesita.

Tipos de grupo en Entra ID

Hay dos ejes distintos que se combinan:

EjeTipoQué hace
FunciónGrupo de seguridadDa o quita permisos de acceso a recursos (archivos, aplicaciones, roles)
Grupo Microsoft 365Pensado principalmente para colaboración: buzón de correo, calendario, archivos compartidos. Pero también puede usarse para dar acceso a recursos (por ejemplo, asignarle permisos sobre un sitio de SharePoint o una aplicación), igual que un grupo de seguridad — solo que además trae de serie las herramientas de colaboración.
MembresíaAsignadaUn administrador añade y quita miembros manualmente, uno a uno
DinámicaSe define una regla (ej. "departamento = Ventas") y Entra ID añade/quita miembros automáticamente
🎯 Truco de examen: un grupo de seguridad solo sirve para dar permisos. Un grupo de Microsoft 365 puede hacer las dos cosas a la vez: dar permisos y traer consigo herramientas de colaboración. Por eso, si la pregunta dice "puede usarse tanto para colaborar como para dar acceso a recursos", se refiere al grupo de Microsoft 365.
Por qué importa la membresía dinámica: si contratan a alguien nuevo en Ventas y su perfil dice "departamento: Ventas", entra automáticamente en el grupo "Ventas" y hereda sus permisos, sin que un administrador tenga que acordarse de añadirlo a mano.

Punto 2.4

Métodos de autenticación en Entra

Ya vimos en el Módulo 1 la teoría general de MFA, biometría y passwordless. Ahora vemos cómo se llaman y para qué sirven específicamente dentro de Entra ID.

SSPR (Self-Service Password Reset)

Permite que un usuario que ha olvidado su contraseña la restablezca él mismo, sin llamar al departamento de IT. Para poder hacerlo, el usuario tiene que haberse registrado antes con métodos de verificación (un teléfono, un correo alternativo, preguntas de seguridad); cuando olvida la contraseña, usa uno de esos métodos para demostrar que es él y crear una nueva.

Por qué importa para la empresa: reduce muchísimo la carga del equipo de soporte técnico — "se me olvidó la contraseña" es una de las llamadas más frecuentes a IT.

Métodos de MFA disponibles en Entra

Es importante fijarse en que los tres primeros métodos son un segundo factor que se añade a la contraseña — la contraseña sigue haciendo falta igualmente. Los dos últimos son passwordless: sustituyen la contraseña por completo, no la complementan.

MétodoCómo funciona¿Necesita contraseña además?
Notificación push (Microsoft Authenticator)Apruebas o rechazas con un toque en la app del móvilSí — es el segundo factor, tras la contraseña
Código de un solo uso (OTP)Generado por la app o enviado por SMSSí — es el segundo factor, tras la contraseña
Llamada de teléfono automáticaConfirmas la identidad respondiendo la llamadaSí — es el segundo factor, tras la contraseña
Llave de seguridad FIDO2Dispositivo físicoNo — passwordless, sustituye la contraseña
Windows Hello for BusinessBiometría o PIN del propio dispositivoNo — passwordless, sustituye la contraseña
🎯 Truco de examen: "passwordless" no significa "sin ningún factor" — significa que ninguno de los factores usados es una contraseña. FIDO2 y Windows Hello combinan "algo que tienes/eres" sin necesitar nunca escribir una contraseña; los otros tres siguen exigiendo la contraseña como primer factor, y añaden un segundo encima.

Protección de contraseñas en Entra

Además de MFA y SSPR, Entra ID protege las contraseñas de dos formas más:

FunciónQué hace
Lista de contraseñas prohibidas (banned password list)Entra ID mantiene una lista global de contraseñas comunes y fáciles de adivinar (como "Password1" o "Qwerty123"), y no deja que ningún usuario las use, ni siquiera con pequeñas variaciones. Las empresas también pueden añadir sus propias palabras prohibidas (por ejemplo, el nombre de la empresa).
Bloqueo inteligente (smart lockout)Si alguien falla varias veces seguidas al escribir la contraseña, Entra ID bloquea temporalmente esa cuenta para frenar ataques de fuerza bruta — pero es lo bastante "inteligente" como para distinguir los intentos del usuario real (que suele fallar de forma distinta) de los de un atacante automatizado, para no bloquear a la gente real sin necesidad.
🎯 Truco de examen: si la pregunta menciona "impedir contraseñas fáciles de adivinar" → lista de contraseñas prohibidas. Si menciona "frenar intentos repetidos de adivinar una contraseña" → smart lockout.
Punto 2.5

Proveedor de identidad, tokens y SSO

Antes de entrar en acceso condicional y roles, conviene cerrar unos conceptos que Microsoft agrupa bajo "conceptos de identidad" y que se preguntan con frecuencia.

¿Qué es un proveedor de identidad (Identity Provider / IdP)?

Es el sistema que se encarga de verificar quién eres y de emitir una "prueba" de esa verificación que otras aplicaciones pueden aceptar. Entra ID actúa como proveedor de identidad para Microsoft 365, Azure y miles de aplicaciones externas conectadas.

¿Qué es un token?

Cuando te autenticas correctamente, el proveedor de identidad no te da tu contraseña de vuelta a cada aplicación — te da un token: una especie de "pase temporal" digital que demuestra que ya te verificaron, sin tener que enviar tus credenciales una y otra vez. Las aplicaciones confían en ese token en vez de pedirte la contraseña cada vez.

Es como la pulsera de un festival de música: la enseñas una vez en la entrada (te autenticas), te dan una pulsera (el token), y durante el resto del día solo enseñas la pulsera para entrar a cualquier zona — nadie vuelve a pedirte el DNI en cada puesto.
🎯 Truco de examen: la autenticación moderna (OAuth 2.0, OpenID Connect) se basa en tokens, no en reenviar la contraseña. Si una pregunta habla de "un pase temporal que demuestra que ya te autenticaste", es un token.

SSO (repaso y ampliación)

Ya vimos en el Módulo 1 que el SSO permite iniciar sesión una vez y acceder a varias aplicaciones sin repetir el proceso. Técnicamente, esto funciona porque el proveedor de identidad emite un token que las demás aplicaciones federadas con él aceptan como válido.

Punto 2.6

Acceso condicional (Conditional Access)

Es una de las funcionalidades más preguntadas del examen. Es la forma en que Entra ID pone en práctica el principio de Zero Trust "verificar explícitamente" que vimos en el Módulo 1: en vez de una regla fija ("siempre pide MFA" o "nunca pide nada"), el acceso condicional evalúa el contexto de cada inicio de sesión en tiempo real y decide qué hacer.

Cómo funciona: señales → decisión → aplicación

Una política de acceso condicional siempre tiene esta estructura:

  1. Señales (condiciones): quién es el usuario, desde qué ubicación se conecta, qué dispositivo usa, a qué aplicación quiere acceder, qué nivel de riesgo tiene ese inicio de sesión.
  2. Decisión: según esas señales, ¿se permite el acceso, se bloquea, o se pide algo más?
  3. Aplicación (control de acceso): lo que se exige o hace — pedir MFA, exigir que el dispositivo esté registrado y cumpla ciertas normas, bloquear directamente, o forzar un cambio de contraseña.
Piensa en un portero de discoteca con reglas flexibles: si vienes con tu DNI y ya te conoce, te deja pasar directo. Si no le suena tu cara, te pide el DNI otra vez. Si llegas a una hora rarísima o pareces sospechoso, no te deja entrar sin más comprobaciones. El "acceso condicional" es exactamente ese portero, aplicado a inicios de sesión.
🎯 Truco de examen: el acceso condicional actúa después de la autenticación básica (usuario/contraseña) pero antes de conceder el acceso final — es una capa de decisión adicional basada en contexto, no un método de autenticación en sí mismo.

Punto 2.7

Roles de Entra ID y RBAC

Ya vimos en el Módulo 1 qué es RBAC (control de acceso basado en roles) a nivel de concepto. Ahora vemos cómo se aplica concretamente dentro de Entra ID.

Roles de Entra ID (roles de administrador)

Son roles pensados para administrar el propio Entra ID y Microsoft 365 — no para acceder a recursos de negocio, sino para gestionar la plataforma. Algunos ejemplos:

RolQué puede hacer
Administrador global (Global Administrator)Acceso completo a todo Entra ID y los servicios conectados — el rol con más poder, debe asignarse con muchísimo cuidado
Administrador de usuarios (User Administrator)Puede crear, modificar y eliminar cuentas de usuario y restablecer contraseñas, pero no tiene acceso a todo lo demás
Lector global (Global Reader)Puede ver la configuración de todo, pero no puede modificar nada — útil para auditoría

RBAC de Azure (roles de recursos)

Son roles distintos, pensados para dar acceso a recursos de Azure (máquinas virtuales, bases de datos, cuentas de almacenamiento), no a la administración de identidades. Ejemplos: "Lector" (solo puede ver recursos), "Colaborador" (puede crear y modificar recursos, pero no dar permisos a otros), "Propietario" (control total, incluido dar permisos a otros).

🎯 Truco de examen: no confundas los roles de Entra ID (administran identidades y la propia plataforma: usuarios, licencias, políticas) con el RBAC de Azure (da acceso a recursos concretos de Azure: máquinas virtuales, bases de datos, etc.). Son dos sistemas de roles distintos y separados.

Punto 2.9

Gobernanza de identidad (Microsoft Entra ID Governance)

La gobernanza de identidad responde a una pregunta incómoda pero muy real en cualquier empresa: con el tiempo, la gente acumula permisos que ya no necesita (cambia de puesto, termina un proyecto, se va de la empresa) y nadie se acuerda de quitárselos. Entra ID Governance da herramientas para controlar esto de forma organizada.

Entitlement Management (gestión de derechos)

Permite crear "paquetes de acceso": un conjunto predefinido de recursos y permisos que alguien recibe automáticamente al solicitarlo (por ejemplo, "paquete de acceso para nuevo comercial", que da acceso a un CRM, una carpeta compartida y un grupo de Teams, todo de una vez), en vez de pedir cada permiso suelto por separado.

Revisiones de acceso (Access Reviews)

Permiten programar revisiones periódicas donde alguien (el propio usuario, su jefe, o un administrador) confirma si un acceso concreto sigue siendo necesario. Si nadie confirma que sigue haciendo falta, el acceso se puede retirar automáticamente.

🎯 Truco de examen: si la pregunta habla de "dar de una vez un conjunto de permisos predefinido" → entitlement management (paquetes de acceso). Si habla de "comprobar periódicamente si un acceso sigue haciendo falta" → revisiones de acceso.

Punto 2.10

Privileged Identity Management (PIM)

PIM aplica el principio de "privilegios mínimos" (Zero Trust, Módulo 1) a los roles con más poder, como Administrador global. En vez de que alguien tenga ese rol activado permanentemente (lo cual es un riesgo enorme si le roban la cuenta), PIM permite que el rol esté desactivado por defecto, y la persona lo "active" temporalmente solo cuando lo necesita, por un tiempo limitado.

Cómo funciona en la práctica

  1. Un administrador es elegible para un rol (puede activarlo), pero no lo tiene activo todo el tiempo.
  2. Cuando necesita hacer una tarea que requiere ese rol, lo activa — a veces requiriendo justificación, aprobación de otra persona, o MFA adicional.
  3. El rol queda activo solo durante un tiempo limitado (por ejemplo, 4 horas), y después se desactiva automáticamente solo.
Es como una llave maestra de un hotel que normalmente está guardada en la caja fuerte de recepción: el personal de limpieza no la lleva encima todo el día "por si acaso" — la pide cuando la necesita, la usa un rato, y la devuelve. Así, si alguien roba la identidad de un empleado, no consigue automáticamente la llave maestra activa.
🎯 Truco de examen: PIM = acceso Just-In-Time (justo a tiempo) a roles con privilegios. Si la pregunta menciona "activar un rol de administrador solo temporalmente, con aprobación" → PIM.

Punto 2.11

Microsoft Entra ID Protection

Es la funcionalidad que detecta comportamientos sospechosos relacionados con identidades, usando señales e inteligencia de amenazas de Microsoft, y calcula un nivel de riesgo para reaccionar automáticamente.

Dos tipos de riesgo que detecta

TipoQué detectaEjemplo
Riesgo de usuario (User risk)Indicios de que las credenciales de un usuario están comprometidasLa contraseña de un empleado aparece filtrada en una base de datos robada conocida
Riesgo de inicio de sesión (Sign-in risk)Indicios de que un inicio de sesión concreto es sospechoso, aunque la cuenta en sí no esté comprometidaAlguien intenta iniciar sesión desde un país donde el usuario nunca ha estado, minutos después de haberlo hecho desde otro país al otro lado del mundo (viaje "imposible")

Según el nivel de riesgo (bajo, medio, alto), Identity Protection puede combinarse con acceso condicional para reaccionar automáticamente: exigir MFA adicional, forzar un cambio de contraseña, o bloquear el acceso directamente.

🎯 Truco de examen: "riesgo de usuario" = la cuenta en sí parece comprometida (credenciales filtradas). "Riesgo de inicio de sesión" = este intento concreto de acceso parece sospechoso, aunque la cuenta esté bien. Identity Protection normalmente actúa en conjunto con acceso condicional, no por separado.

Punto 2.12

Microsoft Entra Verified ID

Es un servicio para emitir y verificar credenciales digitales verificables — como un título universitario, un carné profesional, o la prueba de que trabajas en una empresa — en formato digital, que la propia persona controla y puede presentar cuando se le pida, sin que quien lo verifica tenga que llamar a la institución que lo emitió cada vez.

🎯 Truco de examen: Verified ID no es para empleados de tu empresa iniciando sesión (eso es Entra ID normal) — es para emitir y comprobar credenciales digitales de cosas como titulaciones o certificaciones, de forma descentralizada.
Punto 2.13

Global Secure Access e integración con Security Copilot

Global Secure Access

Es la solución de Microsoft de Security Service Edge (SSE) integrada con Entra: en vez de proteger solo "quién entra", protege también el tráfico de red del usuario hacia internet, aplicaciones y recursos privados, aplicando las mismas políticas de identidad y acceso condicional a nivel de red.

Integración con Microsoft Security Copilot

Security Copilot es el asistente de IA de Microsoft para equipos de seguridad. Integrado con Entra, permite a los administradores hacer preguntas en lenguaje natural sobre identidades, riesgos y configuración (por ejemplo, "¿qué usuarios tienen el rol de Administrador global?") y obtener respuestas o sugerencias sin tener que navegar manualmente por varios paneles.

🎯 Truco de examen: estos dos son los conceptos más nuevos del temario de Entra — basta con reconocer qué son (SSE aplicado a identidad, y asistente de IA para consultas de seguridad), sin necesitar detalle técnico profundo.
